Clase 5 · Laboratorio

Alertas accionables, auto-remediación y visualización unificada

Vas a cerrar el stack completo de observabilidad: alarmas de métrica, compuesta y por detección de anomalía; una regla de EventBridge que dispara un runbook de SSM Automation para auto-remediar; notificaciones por SNS; y un dashboard unificado. Después vas a simular un incidente de CPU y seguir toda la cadena de respuesta.

⏱ Duración · 40–55 min 💵 Costo · < USD 1 (si limpiás al terminar) 📊 Nivel · Intermedio / Avanzado 🌎 Región sugerida · us-east-1
CloudWatch Alarms EventBridge SSM Automation SNS Dashboards CloudFormation

Arquitectura

La plantilla arma el cierre end-to-end: una instancia EC2 genera métricas; tres alarmas encadenadas detectan el incidente; y la alarma compuesta dispara dos reacciones en paralelo — notificar por SNS y auto-remediar vía EventBridge + SSM Automation. Todo se ve en un dashboard unificado.

Flujo: EC2 (CPU alta) → alarma de métrica + alarma de anomalía → alarma compuesta → (a) SNS → email y (b) EventBridge → SSM Automation (auto-remediación) → todo visible en el dashboard unificado.

Requisitos previos

  • Cuenta de AWS con permisos para crear los recursos del laboratorio (EC2, IAM, CloudWatch, SNS, EventBridge, SSM).
  • Una dirección de email válida: SNS te mandará un correo de confirmación de la suscripción.
  • Acceso a la consola web de AWS (no se requiere CLI, pero se muestra la alternativa).
  • Como la plantilla nombra roles IAM, si desplegás por CLI necesitás --capabilities CAPABILITY_NAMED_IAM.
Atención al costoTodos los recursos son de bajo costo, pero recordá ejecutar la limpieza al terminar para no dejar nada corriendo.

Desplegar el stack

Iniciar sesión y elegir la región

Ingresá a la consola de AWS y seleccioná la región del laboratorio.

Consola AWS› selector de región (arriba a la derecha) › US East (N. Virginia) us-east-1

Abrir CloudFormation y crear el stack

CloudFormation› Stacks › Create stack › With new resources

Subí el archivo template.yaml de esta carpeta en Upload a template file.

Nombre y parámetros

Poné un nombre al stack (por ejemplo obs-clase-5) y, sobre todo, completá NotificationEmail con tu email. Revisá el umbral de CPU (CpuAlarmThreshold, 70 % por defecto).

Alternativa por CLISi preferís la línea de comandos:
bash · desplegar por CLI
aws cloudformation deploy \
  --stack-name obs-clase-5 \
  --template-file template.yaml \
  --parameter-overrides NotificationEmail=tu-email@ejemplo.com \
  --capabilities CAPABILITY_NAMED_IAM \
  --region us-east-1

Confirmar y crear

Aceptá las capacidades (IAM) y presioná Submit. Esperá el estado CREATE_COMPLETE.

Checkpoint
El stack aparece en estado CREATE_COMPLETE y la pestaña Resources lista las alarmas, el tópico SNS, el runbook, la regla de EventBridge y el dashboard.

Confirmar la suscripción de email

SNS envía un correo de AWS Notifications a la dirección que pusiste. Abrilo y hacé clic en Confirm subscription. Sin esto no vas a recibir las alertas.

SNS› Topics › <stack>-alertas › Subscriptions
bash · verificar la suscripción
aws sns list-subscriptions-by-topic \
  --topic-arn <AlertTopicArn> \
  --region us-east-1
Checkpoint
La suscripción figura como Confirmed (no PendingConfirmation).

Verificar en la consola

Revisar las tres alarmas

CloudWatch› Alarms › All alarms

Deben existir …-cpu-alta (métrica), …-cpu-anomalia (anomalía) y …-cpu-compuesta (compuesta). Recién creadas suelen estar en OK o INSUFFICIENT_DATA.

bash · listar alarmas
aws cloudwatch describe-alarms \
  --alarm-names obs-clase-5-cpu-alta obs-clase-5-cpu-anomalia obs-clase-5-cpu-compuesta \
  --region us-east-1
Checkpoint
Las tres alarmas aparecen listadas; la compuesta muestra la regla ALARM(cpu-alta) AND ALARM(cpu-anomalia).

Abrir el dashboard unificado

CloudWatch› Dashboards › <stack>-unificado

Vas a ver widgets de CPU (con banda de anomalía), memoria, estado de las alarmas y un widget de logs con las líneas de la app. El enlace directo está en el Output DashboardUrl.

Checkpoint
El widget de logs muestra líneas INFO heartbeat… (esperá 2–3 min si aún está vacío).

Confirmar el runbook y la regla

Systems Manager› Documents › <stack>-remediar-cpu
EventBridge› Rules › <stack>-alarma-a-runbook

La regla debe estar Enabled, con patrón de CloudWatch Alarm State Change → ALARM sobre la alarma compuesta, y el runbook de SSM Automation como target.

Checkpoint
El target de la regla es automation-definition/<stack>-remediar-cpu.

Escenario de investigación

Simulá un incidente de alto uso de CPU y seguí toda la cadena: alarma → EventBridge → runbook de SSM que remedia automáticamente; revisá el dashboard y la notificación por SNS.

Provocar el incidente

Ejecutá el script de estrés dentro de la instancia. Por consola, conectate primero por Session Manager; por CLI, disparalo con send-command (ver el Output SimularIncidenteCommand).

Systems Manager› Session Manager › Start session › elegí la instancia obs-clase-5-ec2
bash · dentro de la instancia (Session Manager)
/usr/local/bin/simular-incidente.sh &
bash · alternativa por CLI (send-command)
aws ssm send-command --document-name AWS-RunShellScript \
  --targets Key=InstanceIds,Values=<InstanceId> \
  --parameters 'commands=["/usr/local/bin/simular-incidente.sh &"]' \
  --region us-east-1

Seguir la propagación de la alarma

En CloudWatch › Alarms, mirá cómo saltan en cadena: primero …-cpu-alta, luego (con historia) …-cpu-anomalia, y cuando ambas están en ALARM, la …-cpu-compuesta.

Checkpoint
La alarma …-cpu-compuesta pasa a ALARM (2–3 min después de iniciar la carga).

Confirmar las dos reacciones

La transición a ALARM dispara en paralelo: (a) el email por SNS y (b) el runbook por EventBridge.

bash · ver la ejecución del runbook
aws ssm describe-automation-executions \
  --filters Key=DocumentNamePrefix,Values=obs-clase-5-remediar-cpu \
  --query "AutomationExecutionMetadataList[0].[AutomationExecutionStatus,ExecutionStartTime]" \
  --region us-east-1
Checkpoint
Recibís el email de la alarma y hay una ejecución de Automation en estado Success.

Verificar la remediación e investigar en el dashboard

El runbook detiene la carga (pkill stress-ng) y escribe ACTION auto-remediacion… en el log. La CPU baja, la compuesta vuelve a OK y SNS envía el aviso de recuperación.

CloudWatch› Logs › Logs Insights › log group /observabilidad/<stack>/app
Logs Insights · incidente y remediación
fields @timestamp, @message
| filter @message like /incidente|auto-remediacion/
| sort @timestamp desc
| limit 50
Checkpoint
En el dashboard ves el pico y su descenso; en los logs, las líneas WARN incidente seguidas de ACTION auto-remediacion.

Documentar el runbook operativo

Dejá escrito, como práctica de producción, la secuencia: detección (alarma compuesta) → notificación (SNS al on-call) → respuesta automática (EventBridge → SSM) → verificación (dashboard + Logs Insights) → escalamiento si no baja → post-incidente (revisar umbrales y el AND de la compuesta).

Explorar y experimentar

Ideas para profundizar y entender el comportamiento de los servicios.

  • Cambiá el AlarmRule de la compuesta de AND a OR y compará la latencia de detección contra el ruido.
  • Bajá CpuAlarmThreshold (por ejemplo a 40 %) y observá cómo dispara antes.
  • Ampliá o achicá la banda de anomalía cambiando el 2 en ANOMALY_DETECTION_BAND(m1, 2).
  • Sumá un segundo target a EventBridge: además del runbook, notificá a Slack/PagerDuty vía SNS, Lambda o AWS Chatbot.
  • Extendé el runbook con un paso aws:changeInstanceState para una remediación más agresiva (reinicio de la instancia).

Troubleshooting

Problemas frecuentes y cómo resolverlos. El detalle completo está en troubleshooting.md.

No llega el email de las alarmas

Síntoma: la alarma pasa a ALARM pero no recibís nada.

Solución: la suscripción SNS quedó PendingConfirmation. Revisá spam, abrí el email de AWS Notifications y hacé clic en Confirm subscription.

La alarma compuesta no pasa a ALARM aunque la CPU está alta

Síntoma: …-cpu-alta está en ALARM pero la compuesta sigue en OK.

Solución: la regla es AND con la de anomalía, que necesita historia para entrenar la banda. Es intencional (reduce ruido). Para forzarla, cambiá el AlarmRule a solo la de métrica.

El runbook no se ejecuta al saltar la alarma

Síntoma: la compuesta está en ALARM pero no hay ejecución de Automation.

Solución: la regla apunta al ARN de la compuesta; verificá que esa fue la que saltó y que la instancia figure como Managed en SSM Fleet Manager.

El widget de logs del dashboard aparece vacío

Síntoma: el widget de tipo log no muestra filas.

Solución: esperá 2–3 min a que el agente empiece a enviar /var/log/obs-app.log; el cron de latido escribe una línea por minuto.

El stack queda en ROLLBACK_COMPLETE

Síntoma: la creación falla y el stack se revierte.

Solución: email inválido en NotificationEmail o falta CAPABILITY_NAMED_IAM. Abrí Events, buscá el primer error (arriba del rollback) y corregí.

Limpieza

Eliminar el stack

La plantilla no usa DeletionPolicy: Retain, así que borrar el stack elimina todo (incluidas las alarmas, el tópico SNS, el runbook, la regla de EventBridge, el dashboard y el log group). No hay bucket S3 que vaciar en este lab.

CloudFormation› Stacks › obs-clase-5 › Delete
bash · eliminar por CLI
aws cloudformation delete-stack --stack-name obs-clase-5 --region us-east-1
Checkpoint
El stack desaparece de la lista (estado DELETE_COMPLETE). No quedan recursos facturables.
