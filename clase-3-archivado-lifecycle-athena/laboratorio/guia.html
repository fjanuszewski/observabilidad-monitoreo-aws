Archivado, ciclo de vida y explotación con Athena

Vas a construir un pipeline de archivado económico de logs (CloudWatch Logs → Amazon Data Firehose → S3 particionado por fecha con Lifecycle a Glacier) y después vas a explotar esos logs con SQL en Amazon Athena para resolver una investigación de seguridad: quién abrió un puerto sensible y cuándo. Todo desplegado desde una única plantilla de CloudFormation.

⏱ Duración · 45–60 min 💵 Costo · < USD 1 (si limpiás al terminar) 📊 Nivel · Intermedio 🌎 Región sugerida · us-east-1
CloudWatch Logs Data Firehose S3 Glacier Glue Athena CloudFormation IAM

Arquitectura

La plantilla despliega el pipeline completo de archivado y explotación. Una instancia EC2 genera eventos con forma CloudTrail (incluido uno de AuthorizeSecurityGroupIngress abriendo el puerto 22) para que el pipeline tenga datos reales que archivar y consultar.

RecursoServicioPara qué
Log group de origenCloudWatch LogsRecibe los eventos antes de archivarlos
Subscription filterCloudWatch LogsEmpuja los eventos a Firehose en near real-time
Delivery streamAmazon Data FirehoseDescomprime, bufferea y escribe en S3 por fecha
Bucket + LifecycleS3 / GlacierData lake con transición a Glacier tras N días
Database + tablaAWS Glue Data CatalogDescribe los datos en S3 (JSON SerDe + projection)
Workgroup + named queryAmazon AthenaEjecuta la consulta SQL de investigación
Instancia t3.microEC2Genera los eventos de ejemplo
Roles IAMIAMEnlazan cada salto con least privilege
FlujoEC2 → CloudWatch Logs → (subscription filter) → Firehose → S3 (yyyy/MM/dd, Lifecycle→Glacier) → Glue Data Catalog → Athena.

Requisitos previos

  • Cuenta de AWS con permisos para crear los recursos del laboratorio (S3, Firehose, Logs, Glue, Athena, EC2, IAM).
  • Acceso a la consola web de AWS (no se requiere CLI, pero se muestra la alternativa).
  • La plantilla nombra roles IAM: al desplegar hay que aceptar CAPABILITY_NAMED_IAM.
Atención al costoTodos los recursos son de bajo costo, pero el bucket S3 queda con objetos. Recordá vaciarlo y borrar el stack al terminar para no dejar nada facturable.

Desplegar el stack

Iniciar sesión y elegir la región

Ingresá a la consola de AWS y seleccioná la región del laboratorio.

Consola AWS› selector de región (arriba a la derecha) › US East (N. Virginia) us-east-1

Abrir CloudFormation y crear el stack

CloudFormation› Stacks › Create stack › With new resources

Subí el archivo template.yaml de esta carpeta en Upload a template file.

Nombre y parámetros

Poné un nombre al stack (por ejemplo obs-clase-3) y revisá los parámetros: NombrePrefijo (único en la cuenta), DiasHastaGlacier (por defecto 30) y DiasRetencionLogGroup.

Alternativa por CLISi preferís la línea de comandos:
bash · desplegar por CLI
aws cloudformation deploy \
  --stack-name obs-clase-3 \
  --template-file template.yaml \
  --parameter-overrides NombrePrefijo=obs-clase3 DiasHastaGlacier=30 \
  --capabilities CAPABILITY_NAMED_IAM \
  --region us-east-1

Confirmar y crear

Marcá la casilla que reconoce que CloudFormation puede crear recursos IAM con nombre (CAPABILITY_NAMED_IAM) y presioná Submit. Esperá el estado CREATE_COMPLETE.

Checkpoint
El stack aparece en estado CREATE_COMPLETE y la pestaña Outputs muestra el nombre del bucket, el delivery stream, la base de Glue y el workgroup de Athena.

Verificar en la consola

Confirmar que llegan eventos al log group

CloudWatch› Log groups › /<prefijo>/cloudtrail-eventos

Deberías ver un log stream generador-… con un par de eventos JSON (uno de ellos, AuthorizeSecurityGroupIngress abriendo el puerto 22).

Checkpoint
El log group tiene al menos un stream con eventos con forma CloudTrail.

Ver los objetos archivados en S3

S3› <prefijo>-archivo-… › prefijo cloudtrail/

Firehose bufferea ~1–2 min antes de escribir. Al abrir, vas a ver la estructura particionada por fecha cloudtrail/AAAA/MM/DD/.

bash · listar objetos archivados
aws s3 ls s3://<prefijo>-archivo-<accountId>-us-east-1/cloudtrail/ --recursive
Checkpoint
Aparecen objetos bajo cloudtrail/AAAA/MM/DD/. Si tarda, esperá el intervalo de buffering (60 s).

Revisar la regla de Lifecycle

S3› bucket › Management › Lifecycle rules

Confirmá la regla archivar-a-glacier: transiciona el prefijo cloudtrail/ a la clase Glacier tras DiasHastaGlacier días.

Checkpoint
La regla existe, está Enabled y apunta a Glacier.

Confirmar el catálogo de Glue y el workgroup de Athena

Athena› Query editor › Workgroup <prefijo>-wg › Database <prefijo>_archivo_db

En el panel izquierdo deberías ver la tabla cloudtrail_eventos. Seleccioná el workgroup del lab (tiene la result location y el límite de bytes escaneados ya configurados).

Checkpoint
Ves la tabla cloudtrail_eventos y el workgroup seleccionado. Athena está listo para consultar.

Investigación con Athena

Escenario: sospechamos que alguien abrió un puerto sensible (22 o 3389) en un security group. Vamos a responder quién lo hizo y cuándo, consultando los eventos de CloudTrail archivados en S3.

Abrir la consulta guardada

Athena› Query editor › pestaña Saved queries › <prefijo>-quien-abrio-puerto

La plantilla creó una named query lista para ejecutar. También podés pegarla a mano en el editor.

Ejecutar la consulta de investigación

Ajustá la partición de fecha (anio/mes) a la fecha en la que corriste el lab y ejecutá:

sql · quién abrió un puerto sensible
SELECT
  eventtime,
  useridentity.arn      AS quien,
  useridentity.username AS usuario,
  sourceipaddress       AS ip_origen,
  awsregion             AS region,
  requestparameters     AS parametros
FROM "<prefijo>_archivo_db"."cloudtrail_eventos"
WHERE eventname = 'AuthorizeSecurityGroupIngress'
  AND (
        requestparameters LIKE '%"fromPort":22%'
     OR requestparameters LIKE '%"fromPort":3389%'
  )
  AND anio = '2026' AND mes = '07'   -- ajustá a la fecha de tus datos
ORDER BY eventtime DESC;
Por qué el WHERE de fechaFiltrar por las columnas de partición (anio/mes/dia) hace que Athena escanee menos datos: más rápido y más barato. El workgroup además corta cada query a 100 MB.
Checkpoint
La consulta devuelve una fila con pedro.admin, su IP de origen (203.0.113.24), la región y el eventtime. Ya sabés quién abrió el puerto 22 y cuándo.

(Opcional) Ejecutar la investigación por CLI

bash · lanzar la query por CLI
aws athena start-query-execution \
  --work-group <prefijo>-wg \
  --query-string "SELECT eventtime, useridentity.arn, sourceipaddress \
    FROM \"<prefijo>_archivo_db\".\"cloudtrail_eventos\" \
    WHERE eventname='AuthorizeSecurityGroupIngress' \
      AND anio='2026' AND mes='07';" \
  --region us-east-1

Los resultados quedan en el prefijo athena-results/ del bucket.

Explorar y experimentar

Ideas para profundizar y entender el comportamiento del pipeline.

  • Cambiá el buffering de Firehose (SizeInMBs / IntervalInSeconds) y observá cómo varía la cantidad y el tamaño de los objetos en S3. Menos objetos grandes = mejor para Glacier.
  • Bajá DiasHastaGlacier a 1 en un stack de prueba y mirá cómo la Lifecycle transiciona los objetos (y por qué Athena luego no los lee sin restaurar).
  • Quitá el filtro de partición de la query y compará los bytes escaneados: el workgroup debería cortar a 100 MB.
  • Buscá RevokeSecurityGroupIngress para ver quién cerró el puerto y correlacionar con la apertura.
  • Compará cuándo usar Logs Insights (lo reciente en CloudWatch Logs) vs Athena (lo archivado en S3, gran volumen, bajo costo).

Troubleshooting

Problemas frecuentes y cómo resolverlos. Ideal para practicar diagnóstico. Detalle completo en troubleshooting.md.

Athena devuelve basura o caracteres ilegibles

Síntoma: los objetos en S3 no se leen bien.

Solución: los logs de CloudWatch llegan gzip-comprimidos; hay que habilitar la decompresión en Firehose (esta plantilla ya la trae con Decompression + CloudWatchLogProcessing).

La consulta de Athena devuelve cero filas

Síntoma: hay objetos en S3 pero la query no trae nada.

Solución: verificá que anio/mes/dia coincidan con la fecha real de los objetos, que el LOCATION apunte a cloudtrail/ y que la partition projection cubra el rango de fechas.

La query es lenta o cara

Síntoma: Athena escanea demasiados datos.

Solución: incluí siempre un WHERE sobre las columnas de partición de fecha. El workgroup corta cada query a 100 MB.

Athena falla al leer objetos ya archivados en Glacier

Síntoma: error sobre objetos en clase Glacier.

Solución: Athena no lee Glacier no-instant sin restaurar; solo se archivan datos ya "fríos" (tras DiasHastaGlacier días).

El stack queda en ROLLBACK_COMPLETE

Síntoma: la creación falla y se revierte.

Solución: suele ser el nombre de bucket ya en uso (son globales). Abrí Events, buscá el primer error y cambiá NombrePrefijo.

Limpieza

Vaciar el bucket S3

Firehose y Athena dejan objetos en el bucket, y un bucket con contenido no se puede borrar. Vacialo primero.

S3› <prefijo>-archivo-… › Empty
bash · vaciar el bucket
aws s3 rm s3://<prefijo>-archivo-<accountId>-us-east-1/ --recursive
Checkpoint
El bucket queda vacío (sin objetos ni versiones).

Eliminar el stack

CloudFormation› Stacks › obs-clase-3 › Delete
bash · eliminar por CLI
aws cloudformation delete-stack --stack-name obs-clase-3 --region us-east-1
Checkpoint
El stack desaparece de la lista (estado DELETE_COMPLETE). No quedan recursos facturables.
CostoObjetivo del lab: < USD 1. EC2 t3.micro + gp3 de 8 GB por los minutos que dure, Firehose y S3/Glacier por unos KB de ejemplo, y Athena por datos escaneados (acotado por el workgroup).
