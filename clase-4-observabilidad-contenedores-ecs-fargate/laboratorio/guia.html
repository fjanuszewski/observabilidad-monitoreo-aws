Clase 4 · Laboratorio

Observabilidad de contenedores (ECS + Fargate)

Vas a desplegar un clúster de Amazon ECS con un servicio en AWS Fargate, con Container Insights (enhanced observability) habilitado y los logs del contenedor ruteados con FireLens (Fluent Bit) hacia CloudWatch Logs. Después vas a explorar las métricas por tarea y contenedor, forzar un reinicio y correlacionar el pico de métricas con los logs para hallar la causa.

⏱ Duración · 30–45 min 💵 Costo · < USD 1 (si limpiás al terminar) 📊 Nivel · Intermedio 🌎 Región sugerida · us-east-1
Amazon ECS AWS Fargate Container Insights FireLens CloudWatch CloudFormation

Arquitectura

La plantilla despliega todo el laboratorio en una sola pasada: la red mínima, el clúster con Container Insights enhanced, la task definition con la app y el sidecar FireLens, y el servicio Fargate.

RecursoServicioPara qué
VPC + subred pública + IGWAmazon VPCRed single-AZ para que la tarea baje las imágenes públicas (sin NAT)
Clúster <prefijo>-clusterAmazon ECSClúster con Container Insights enhanced
Task definition (app + log_router)ECS / FargateContenedor de app (httpd + logs a stdout) + sidecar FireLens
Servicio <prefijo>-svcAmazon ECSMantiene 1 tarea Fargate corriendo
Log group /<prefijo>/appCloudWatch LogsDestino final de los logs de la app (los escribe FireLens)
Log group /<prefijo>/firelensCloudWatch LogsDiagnóstico del propio sidecar Fluent Bit
Execution role + Task roleIAMPull de imagen + logs (mínimos)

Flujo: Servicio Fargate → Task → contenedor app → log driver awsfirelens → sidecar Fluent Bit → CloudWatch Logs. En paralelo, Container Insights (enhanced) publica las métricas por clúster, servicio, tarea y contenedor.

Fargate no expone el hostAl no tener acceso a la instancia EC2 subyacente, no administrás ni parcheás hosts, y no hay parámetro de AMI. Las métricas de tarea/contenedor las provee Container Insights, no un agente en el host.

Requisitos previos

  • Cuenta de AWS con permisos para crear ECS, Fargate, VPC, IAM y CloudWatch Logs.
  • Acceso a la consola web de AWS (no se requiere CLI, pero se muestra la alternativa).
  • Para la vía CLI: AWS CLI v2 configurado. La plantilla nombra roles IAM, así que hay que aceptar la capacidad CAPABILITY_NAMED_IAM.
Atención al costoContainer Insights enhanced factura por observaciones y la tarea Fargate por hora. Todo es de bajo costo, pero recordá ejecutar la limpieza al terminar para no dejar nada corriendo.

Desplegar el stack

Iniciar sesión y elegir la región

Ingresá a la consola de AWS y seleccioná la región del laboratorio.

Consola AWS› selector de región (arriba a la derecha) › US East (N. Virginia) us-east-1

Abrir CloudFormation y crear el stack

CloudFormation› Stacks › Create stack › With new resources

Subí el archivo template.yaml de esta carpeta en Upload a template file.

Nombre y parámetros

Poné un nombre al stack (por ejemplo obs-clase-4) y revisá los parámetros: prefijo de nombres, CIDRs de la red, imágenes de la app y del sidecar, y la retención de logs. Los valores por defecto ya sirven para el lab.

Alternativa por CLISi preferís la línea de comandos:
bash · desplegar por CLI
aws cloudformation deploy \
  --stack-name obs-clase-4 \
  --template-file template.yaml \
  --parameter-overrides file://parameters.example.json \
  --capabilities CAPABILITY_NAMED_IAM \
  --region us-east-1

Confirmar y crear

Marcá la casilla de capacidades IAM (CAPABILITY_NAMED_IAM) y presioná Submit. Esperá el estado CREATE_COMPLETE (la creación de la red + clúster + servicio tarda unos minutos).

Checkpoint
El stack aparece en estado CREATE_COMPLETE y la pestaña Outputs muestra el nombre del clúster, del servicio y los log groups.

Verificar en la consola

La tarea está corriendo

Amazon ECS› Clusters › <prefijo>-cluster › Services › <prefijo>-svc › Tasks

Confirmá que hay 1 tarea en estado RUNNING con sus dos contenedores (app y log_router) sanos.

bash · estado del servicio por CLI
aws ecs describe-services \
  --cluster obs-clase4-cluster --services obs-clase4-svc \
  --query "services[0].{status:status,running:runningCount,desired:desiredCount}" \
  --region us-east-1
Checkpoint
runningCount == desiredCount == 1 y el status del servicio es ACTIVE.

Los logs llegan vía FireLens

CloudWatch› Log groups › /<prefijo>/app

Abrí el log group de la app y mirá el stream app-…. Vas a ver líneas JSON INFO y, cada 10 iteraciones, una WARN. Esos logs los emitió el contenedor app y los ruteó el sidecar FireLens (Fluent Bit).

bash · seguir los logs en vivo
aws logs tail /obs-clase4/app --follow --region us-east-1
Checkpoint
Ves eventos JSON con un campo seq incremental. El log group /<prefijo>/firelens muestra el arranque del router sin errores.

Métricas por tarea y contenedor en Container Insights

CloudWatch› Insights › Container Insights › Performance monitoring › ECS Clusters › tu clúster

Cambiá el selector de nivel entre Clusters, Services, Tasks y Containers. Con enhanced observability vas a ver CPU y memoria hasta el nivel de contenedor, además del seguimiento de despliegues.

Checkpoint
En el nivel Tasks/Containers ves puntos de datos de CPU y memoria de tu tarea (pueden tardar 3–5 min en aparecer tras el arranque).

Explorar y experimentar

Ideas para profundizar en la observabilidad de contenedores:

  • Estándar vs enhanced: cambiá el clúster a containerInsights = enabled y observá qué métricas por contenedor desaparecen.
  • awslogs vs FireLens: cambiá el log driver del contenedor app de awsfirelens a awslogs y compará la simplicidad frente a la flexibilidad de rutear a S3 / OpenSearch / terceros con FireLens.
  • Logs Insights: en el log group de la app, contá eventos por level con filter level = "WARN" | stats count() by bin(1m).
  • APM (siguiente paso): sumá un sidecar ADOT y activá CloudWatch Application Signals para empezar a ver trazas correlacionadas (introducción del Módulo 4).

Escenario de investigación

Explorá Container Insights a nivel de tarea y contenedor, forzá un reinicio del contenedor y correlacioná el pico de métricas con los logs para hallar la causa.

Tomá una línea base

En Container Insights, bajá al nivel de Tasks y Containers y anotá el uso normal de CPU y memoria del contenedor app (bajo: es un httpd ocioso que imprime una línea cada 5 s).

Forzá un reinicio de la tarea

Provocá que el servicio recree la tarea para generar un evento observable. Por consola: ECS › Cluster › Servicio › Tasks, seleccioná la tarea y Stop (el servicio, con desiredCount: 1, la vuelve a crear). O por CLI:

bash · parar la tarea (el servicio la recrea)
TASK_ARN=$(aws ecs list-tasks --cluster obs-clase4-cluster \
  --service-name obs-clase4-svc --query "taskArns[0]" \
  --output text --region us-east-1)

aws ecs stop-task --cluster obs-clase4-cluster --task "$TASK_ARN" \
  --reason "lab: forzar reinicio para investigar" --region us-east-1
AlternativaUn aws ecs update-service --force-new-deployment también reemplaza la tarea y dispara el deployment tracking.

Correlacioná el pico con los logs

En Container Insights vas a ver la tarea vieja desaparecer y una tarea nueva arrancar (spike de arranque en CPU/memoria) y el evento de reemplazo en el deployment tracking del servicio. Alineá ese timestamp con los logs: en /<prefijo>/app el campo seq se reinicia desde 1 cuando arranca la tarea nueva.

Checkpoint
El corte de la secuencia en los logs coincide con el spike de arranque en las métricas: esa es la huella del reinicio, y confirma qué tarea/contenedor se reinició y cuándo.
Causa raízEl pico de métricas coincide con el arranque de una tarea nueva; los logs (secuencia reiniciada + los metadatos de ECS que agrega FireLens) confirman el reinicio. Esa es la mecánica de correlación infraestructura ↔ servicio ↔ log que después extendés a trazas con ADOT / Application Signals / X-Ray.

Troubleshooting

Problemas frecuentes y cómo resolverlos. Ideal para practicar diagnóstico. Ver el troubleshooting completo.

La tarea no arranca o entra en loop de reinicio

Síntoma: el servicio nunca llega a runningCount == 1 o la tarea se detiene una y otra vez.

Solución: revisá el Stopped reason de la tarea. Suele ser CannotPullContainerError (sin salida a Internet: falta IP pública o ruta al IGW) o falta de permisos del execution role.

No aparecen logs de la app en CloudWatch

Síntoma: el log group /<prefijo>/app está vacío aunque la tarea corre.

Solución: verificá que el task role tenga logs:CreateLogStream y logs:PutLogEvents sobre ese log group, que el log group exista (o poné auto_create_group: "true") y que el sidecar log_router esté sano (mirá /<prefijo>/firelens).

No hay métricas en Container Insights

Síntoma: el clúster no muestra datos.

Solución: confirmá que el clúster tenga containerInsights = enhanced, esperá 3–5 min tras el arranque y asegurate de estar mirando el nivel correcto (Tasks/Containers para ver CPU/memoria por contenedor).

El stack queda en ROLLBACK_COMPLETE

Síntoma: la creación falla y el stack se revierte.

Solución: abrí la pestaña Events, buscá el primer error (arriba del rollback). Suele ser colisión de nombres IAM (cambiá NombrePrefijo), CIDR inválido o falta de CAPABILITY_NAMED_IAM.

Limpieza

Eliminar el stack

CloudFormation› Stacks › obs-clase-4 › Delete

Al borrar el stack se baja el servicio, la tarea y toda la red. No hay bucket S3 ni objetos que vaciar en este lab, y la plantilla no usa DeletionPolicy: Retain, así que todo se elimina.

bash · eliminar por CLI
aws cloudformation delete-stack --stack-name obs-clase-4 --region us-east-1
Checkpoint
El stack desaparece de la lista (estado DELETE_COMPLETE). El clúster ya no aparece en ECS y los log groups se borran. No quedan recursos facturables.
