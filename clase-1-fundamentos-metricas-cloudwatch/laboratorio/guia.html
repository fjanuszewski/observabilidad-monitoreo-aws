Fundamentos, métricas y monitoreo de EC2 con CloudWatch

Vas a desplegar con CloudFormation la base del entorno de observabilidad: una VPC con subred pública y una instancia EC2 t3.micro que instala el CloudWatch agent (con su configuración versionada en SSM Parameter Store), publica métricas del sistema operativo y una métrica personalizada con PutMetricData. Después vas a verificar los datos en CloudWatch y a diagnosticar, por SSM Session Manager, por qué dejan de llegar métricas cuando el agente se detiene.

⏱ Duración · 40–55 min 💵 Costo · < USD 1 (si limpiás al terminar) 📊 Nivel · Intermedio 🌎 Región sugerida · us-east-1
CloudFormation VPC EC2 CloudWatch Systems Manager IAM

Arquitectura

La plantilla template.yaml despliega todo el entorno como un único stack. La instancia expone dos rutas de métricas hacia CloudWatch: las del hipervisor (namespace AWS/EC2, sin agente) y las del sistema operativo (namespace CWAgent, publicadas por el agente), más una métrica personalizada enviada con PutMetricData.

RecursoServicioPara qué
VPC + subred pública + IGWVPCRed con salida a internet para el agente y SSM
IAM Role + Instance ProfileIAMPermisos sin credenciales estáticas (CloudWatchAgentServerPolicy + AmazonSSMManagedInstanceCore)
Parámetro con la config JSONSSM Parameter StoreConfiguración versionada del CloudWatch agent
Instancia t3.microEC2Host monitoreado; UserData instala y arranca el agente
Namespaces CWAgent / Curso/ObsCloudWatchReciben métricas del SO y la métrica personalizada

Requisitos previos

  • Cuenta de AWS con permisos para crear VPC, EC2, IAM, SSM y CloudWatch.
  • Acceso a la consola web de AWS. No se requiere CLI, pero se muestra la alternativa en cada paso.
  • Para la vía CLI: aws-cli configurado (aws configure) con credenciales de la cuenta.
  • El stack crea un rol IAM con nombre generado, así que al desplegar hay que aceptar la capacidad CAPABILITY_IAM.
Atención al costoTodos los recursos son de bajo costo (t3.micro, disco gp3 de 8 GiB, single-AZ), pero recordá ejecutar la limpieza al terminar para no dejar la instancia corriendo.

Desplegar el stack

Iniciar sesión y elegir la región

Ingresá a la consola de AWS y seleccioná la región del laboratorio.

Consola AWS› selector de región (arriba a la derecha) › US East (N. Virginia) us-east-1

Abrir CloudFormation y crear el stack

CloudFormation› Stacks › Create stack › With new resources (standard)

En Specify template elegí Upload a template file y subí el archivo template.yaml de esta carpeta.

Nombre y parámetros

Poné un nombre al stack (por ejemplo obs-clase-1). Los parámetros ya traen defaults de bajo costo; podés dejarlos como están. La AMI se resuelve automáticamente desde SSM, no hay que elegir un ID.

Alternativa por CLIDesplegá el stack completo con un solo comando (desde esta carpeta):
bash · desplegar por CLI
aws cloudformation deploy \
  --stack-name obs-clase-1 \
  --template-file template.yaml \
  --parameter-overrides file://parameters.example.json \
  --capabilities CAPABILITY_IAM \
  --region us-east-1
Notadeploy acepta los parámetros con --parameter-overrides Key=Value; el archivo parameters.example.json también sirve tal cual con create-stack usando --parameters file://parameters.example.json.

Confirmar y crear

En la última pantalla marcá I acknowledge that AWS CloudFormation might create IAM resources y presioná Submit. Esperá el estado CREATE_COMPLETE (2–4 minutos).

Checkpoint
El stack aparece en CREATE_COMPLETE y la pestaña Resources lista la VPC, la subred, el IGW, el rol IAM, el parámetro SSM y la instancia EC2.

Anotar los Outputs

CloudFormation› Stacks › obs-clase-1 › Outputs

Copiá el InstanceId y los comandos de verificación (ListSoMetricsCommand, SessionManagerCommand). Los vas a usar en las secciones siguientes.

Verificar en la consola

Esperar a que el agente publique

Tras CREATE_COMPLETE, el UserData instala y arranca el agente. Las primeras métricas del SO tardan unos 3–5 minutos en aparecer.

Ver las métricas del sistema operativo (CWAgent)

CloudWatch› Metrics › All metrics › CWAgent › InstanceId

Confirmá que aparecen mem_used_percent y disk_used_percent con datos para tu InstanceId.

bash · listar métricas del SO por CLI
aws cloudwatch list-metrics \
  --namespace CWAgent \
  --dimensions Name=InstanceId,Value=<INSTANCE_ID> \
  --region us-east-1
Checkpoint
El namespace CWAgent muestra mem_used_percent y disk_used_percent con puntos de datos.

Ver las métricas del hipervisor (AWS/EC2)

CloudWatch› Metrics › All metrics › EC2 › Per-Instance Metrics

Buscá CPUUtilization para tu instancia. Estas métricas las emite EC2 sin agente.

bash · listar CPUUtilization por CLI
aws cloudwatch list-metrics \
  --namespace AWS/EC2 --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=<INSTANCE_ID> \
  --region us-east-1
Checkpoint
Ves CPUUtilization en AWS/EC2. Nota que ahí NO existen mem_used_percent ni disk_used_percent: el hipervisor no ve dentro del SO.

Ver la métrica personalizada (Curso/Obs)

CloudWatch› Metrics › All metrics › Curso/Obs

El UserData publica TareasCompletadas con PutMetricData y un cron la refresca cada minuto.

bash · listar la métrica personalizada
aws cloudwatch list-metrics \
  --namespace Curso/Obs \
  --dimensions Name=InstanceId,Value=<INSTANCE_ID> \
  --region us-east-1
Checkpoint
El namespace Curso/Obs muestra TareasCompletadas con datos.

Explorar y experimentar

Ideas para profundizar y entender el comportamiento de las métricas:

  • Poné mem_used_percent y CPUUtilization en el mismo gráfico y compará granularidad y origen (SO vs hipervisor).
  • Generá carga de CPU en la instancia y observá subir CPUUtilization (ver el comando en el escenario de investigación).
  • Probá Metrics Insights: en la pestaña Query, ejecutá SELECT AVG(mem_used_percent) FROM CWAgent GROUP BY InstanceId.
  • Publicá tu propia métrica con PutMetricData desde tu máquina y verificala en Curso/Obs.
  • Revisá la config del agente en SSM Parameter Store y cambiá metrics_collection_interval a 10 s (alta resolución) para ver el efecto.

Escenario de investigación

Objetivo: comparar las métricas del SO (mem_used_percent, disk_used_percent) con las del hipervisor (CPUUtilization) y luego diagnosticar por qué dejan de llegar las métricas del SO cuando el agente se detiene, usando SSM Session Manager y los logs del agente.

Conectarse a la instancia sin SSH

Systems Manager› Session Manager › Start session › elegí la instancia obs-clase-1-ec2
bash · Session Manager por CLI
aws ssm start-session --target <INSTANCE_ID> --region us-east-1
Checkpoint
Obtenés una shell en la instancia sin abrir el puerto 22 ni usar llaves SSH.

Confirmar que el agente está corriendo

bash · estado del agente
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl -a status

El campo status debe decir running.

Detener el agente y observar el efecto

bash · detener el agente
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl -a stop

Volvé a CloudWatch y mirá los gráficos de mem_used_percent y CPUUtilization lado a lado durante 5–10 minutos.

Checkpoint
Las métricas del SO (CWAgent) dejan de recibir puntos nuevos, mientras que CPUUtilization (AWS/EC2) sigue llegando: el hipervisor no depende del agente.

Diagnosticar con los logs del agente

bash · leer el log del agente
sudo tail -n 50 /opt/aws/amazon-cloudwatch-agent/logs/amazon-cloudwatch-agent.log

El log confirma que el agente fue detenido. Si en vez de detenerlo vos, no arrancó, este log y /var/log/cloud-init-output.log (salida del UserData) muestran la causa (config inválida, permisos, etc.).

Reanudar el agente

bash · volver a arrancar con la config de SSM
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
  -a fetch-config -m ec2 -s \
  -c ssm:/observabilidad/obs-clase-1/cloudwatch-agent-config
Checkpoint
Tras 3–5 minutos, las métricas del SO vuelven a aparecer en CWAgent. Diagnóstico cerrado.

Troubleshooting

Problemas frecuentes y cómo resolverlos. Detalle completo en troubleshooting.md.

El stack queda en ROLLBACK_COMPLETE

Síntoma: la creación falla y el stack se revierte.

Solución: abrí la pestaña Events, buscá el primer error (arriba del rollback) y corregí el parámetro o permiso. Borrá el stack fallido antes de reintentar.

No aparecen métricas del SO en CloudWatch

Síntoma: el namespace CWAgent está vacío.

Solución: esperá 3–5 minutos; conectate por Session Manager y verificá el estado con amazon-cloudwatch-agent-ctl -a status; revisá el rol de la instancia y el log del agente.

Busco mem_used_percent en AWS/EC2 y no aparece

Síntoma: no encontrás memoria ni uso de disco en el namespace del hipervisor.

Solución: es esperado. EC2 nunca emite esas métricas: viven en CWAgent y solo existen con el agente corriendo.

La instancia no aparece en Session Manager

Síntoma: no podés iniciar sesión desde SSM.

Solución: verificá que el rol tenga AmazonSSMManagedInstanceCore y que la subred tenga salida a internet (ruta al IGW). Esperá 2–3 minutos a que el SSM Agent registre la instancia.

Limpieza

Eliminar el stack

CloudFormation› Stacks › obs-clase-1 › Delete

Al borrar el stack se eliminan todos los recursos (instancia, VPC, IGW, rol, parámetro SSM). No hay buckets ni recursos con DeletionPolicy: Retain, así que nada queda facturando.

bash · eliminar por CLI
aws cloudformation delete-stack --stack-name obs-clase-1 --region us-east-1
Métricas personalizadasLas métricas ya publicadas en CloudWatch (CWAgent, Curso/Obs) no se borran con el stack; expiran solas por retención y no generan costo adicional una vez que dejan de recibir datos.
Checkpoint
El stack desaparece de la lista (estado DELETE_COMPLETE). No quedan recursos facturables.
