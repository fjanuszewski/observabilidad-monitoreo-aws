Logs: recolección, lectura y retención

Vas a centralizar los logs de una app que corre en EC2 dentro de Amazon CloudWatch Logs, definir su retención, contar los errores con un metric filter y una alarma, y usar Logs Insights para localizarlos durante un troubleshooting. Todo declarado como código en una plantilla de CloudFormation.

⏱ Duración · 30–45 min 💵 Costo · < USD 1 (si limpiás al terminar) 📊 Nivel · Intermedio 🌎 Región sugerida · us-east-1
CloudWatch Logs CloudWatch Alarms EC2 KMS CloudFormation IAM

Arquitectura

La app en EC2 escribe un archivo de log; el agente unificado de CloudWatch lo publica en un log group con retención definida. Sobre ese log group, un metric filter cuenta las líneas ERROR y alimenta una alarma que (opcionalmente) notifica por SNS. Una consulta de Logs Insights queda guardada para el troubleshooting.

RecursoServicioPara qué
Instancia t3.micro + agenteEC2Genera la app y publica sus logs
Log group con RetentionInDaysCloudWatch LogsAlmacena los logs y define la rotación
Metric filter contar-erroresCloudWatch LogsConvierte líneas ERROR en una métrica
Alarma obs-clase-2-erroresCloudWatch AlarmsPasa a ALARM al superar el umbral
QueryDefinition guardadaCloudWatch LogsConsulta de Logs Insights reutilizable
Topic SNS (opcional)SNSNotifica por email al disparar la alarma
KMS key (opcional)KMSCifra el log group en reposo
Rol e instance profileIAMAutoriza al agente a escribir en Logs
Clase de log groupEl parámetro LogGroupClass permite elegir STANDARD (habilita metric filters, Live Tail y subscription filters) o INFREQUENT_ACCESS (más barato, solo consultable con Logs Insights). El metric filter y la alarma solo se crean con la clase STANDARD.

Requisitos previos

  • Cuenta de AWS con permisos para crear EC2, CloudWatch Logs, IAM, SNS y KMS.
  • Una VPC por defecto en la región (las cuentas nuevas ya la traen); la instancia se lanza ahí sin abrir puertos.
  • Acceso a la consola web de AWS. Se muestra también la vía CLI (opcional).
  • Como se crean roles con nombre, al desplegar hay que aceptar la capacidad CAPABILITY_NAMED_IAM.
Atención al costoTodos los recursos son de bajo costo (una t3.micro y un disco gp3 chico), pero recordá ejecutar la limpieza al terminar para no dejar la instancia corriendo.

Desplegar el stack

Iniciar sesión y elegir la región

Ingresá a la consola de AWS y seleccioná la región del laboratorio.

Consola AWS› selector de región (arriba a la derecha) › US East (N. Virginia) us-east-1

Abrir CloudFormation y crear el stack

CloudFormation› Stacks › Create stack › With new resources

Elegí Upload a template file y subí el archivo template.yaml de esta carpeta.

Nombre y parámetros

Poné un nombre al stack (por ejemplo obs-clase-2) y revisá los parámetros. Los defaults ya sirven; si querés recibir la notificación por email, completá NotificationEmail, y si querés cifrar el log group poné EnableKmsEncryption en true.

Alternativa por CLISi preferís la línea de comandos (usa parameters.example.json de esta carpeta):
bash · desplegar por CLI
aws cloudformation deploy \
  --stack-name obs-clase-2 \
  --template-file template.yaml \
  --parameter-overrides file://parameters.example.json \
  --capabilities CAPABILITY_NAMED_IAM \
  --region us-east-1
Notadeploy también acepta los parámetros inline: --parameter-overrides RetentionInDays=7 NotificationEmail=vos@ejemplo.com.

Confirmar y crear

Aceptá la casilla de capacidades de IAM (CAPABILITY_NAMED_IAM) y presioná Submit. Esperá el estado CREATE_COMPLETE (1–3 minutos).

Checkpoint
El stack aparece en estado CREATE_COMPLETE y la pestaña Resources lista el log group, el metric filter, la alarma, la instancia EC2 y el rol IAM.

Verificar en la consola

Confirmar que llegan los logs

CloudWatch› Logs › Log groups › /obs/clase-2/app

Abrí el log group y entrá al log stream de la instancia. Deberías ver líneas INFO y, cada pocos minutos, líneas ERROR. Revisá también que el log group muestre Retention: 7 days.

bash · ver logs por CLI
aws logs tail /obs/clase-2/app --follow --region us-east-1
Checkpoint
Aparecen eventos en el log stream y la retención figura en 7 días (dales 3–5 minutos tras el CREATE_COMPLETE).

Ver la métrica del metric filter

CloudWatch› Metrics › All metrics › Obs/Clase2 › AppErrorCount

Confirmá que la métrica AppErrorCount existe y publica datos (gracias a DefaultValue: 0 vas a ver ceros incluso sin errores).

Checkpoint
La métrica aparece con puntos de datos y sube cuando la app registra un ERROR.

Revisar la alarma y la consulta guardada

CloudWatch› Alarms › obs-clase-2-errores

La alarma arranca en OK o INSUFFICIENT_DATA y pasa a ALARM cuando la métrica supera el umbral. La consulta guardada está en Logs › Logs Insights › Queries › Obs-Clase2/Errores-por-periodo.

Checkpoint
Ves la alarma en la lista y la consulta guardada disponible para ejecutar.

Escenario de investigación

Objetivo: generar errores en la app, ver la alarma pasar a ALARM y usar Logs Insights para localizar y contar los errores por período — el flujo real de un troubleshooting.

Conectarte a la instancia (Session Manager)

EC2› Instances › obs-clase-2-app › Connect › Session Manager › Connect

No hace falta SSH ni abrir puertos: el rol IAM incluye AmazonSSMManagedInstanceCore. Por CLI:

bash · abrir sesión SSM
aws ssm start-session --target <INSTANCE_ID> --region us-east-1

Inyectar líneas ERROR a mano

Ya dentro de la instancia, escribí varias líneas ERROR en el archivo que el agente vigila para acelerar el disparo de la alarma:

bash · generar errores
for i in $(seq 1 10); do
  echo "$(date -u +%FT%TZ) ERROR request id=demo-$i status=500 msg='forced error'" \
    | sudo tee -a /var/log/app/app.log
done
Qué pasaEl agente publica esas líneas en el log group, el metric filter las cuenta y AppErrorCount sube por encima del umbral.

Ver la alarma pasar a ALARM

CloudWatch› Alarms › obs-clase-2-errores

En 1–2 minutos la alarma cambia a ALARM. Si configuraste NotificationEmail (y confirmaste la suscripción SNS), te llega el correo. Verificá por CLI:

bash · estado de la alarma
aws cloudwatch describe-alarms \
  --alarm-names obs-clase-2-errores-obs-clase-2 \
  --query "MetricAlarms[0].StateValue" --output text \
  --region us-east-1
Checkpoint
La alarma muestra ALARM y (si aplica) llegó la notificación por email.

Localizar y contar los errores con Logs Insights

CloudWatch› Logs › Logs Insights › log group /obs/clase-2/app

Ejecutá la consulta guardada para listar los errores del período, y una variante con stats para contarlos por minuto. Acordate de acotar el rango de tiempo (por ejemplo, últimos 30 min) antes de correrla.

Logs Insights · listar errores
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 100
Logs Insights · contar errores por minuto
fields @timestamp, @message
| filter @message like /ERROR/
| stats count(*) as errores by bin(1m)
| sort errores desc
Checkpoint
La consulta devuelve las líneas ERROR (incluyendo tus forced error) y el stats muestra el conteo por minuto.

(Opcional) Ver los errores en vivo con Live Tail

CloudWatch› Logs › Live Tail › log group /obs/clase-2/app

Iniciá una sesión de Live Tail con el filtro ERROR y volvé a inyectar líneas: vas a verlas aparecer en streaming en tiempo real. Live Tail solo funciona en log groups de clase STANDARD.

Explorar y experimentar

Ideas para profundizar en el pilar de logs:

  • Retención: volvé a desplegar con RetentionInDays=1 o =30 y observá el cambio en el log group. Sin este parámetro la retención sería indefinida (costo creciente).
  • Clase de log: probá LogGroupClass=INFREQUENT_ACCESS y comprobá que el stack ya no crea metric filter ni alarma (esa clase no los soporta), pero sí se puede consultar con Logs Insights.
  • Cifrado KMS: desplegá con EnableKmsEncryption=true y verificá en el log group que aparece la KMS key asociada.
  • Umbral: subí ErrorThreshold a 5 y mirá cuántos errores hacen falta para disparar la alarma.
  • FilterPattern: cambiá el patrón del metric filter (por ejemplo a "status=500") y observá cómo cambia el conteo.

Troubleshooting

Problemas frecuentes y cómo resolverlos. Ideal para practicar diagnóstico. Hay más casos en troubleshooting.md.

La alarma nunca sale de INSUFFICIENT_DATA

Síntoma: la alarma no dispara ni siquiera con errores.

Solución: el metric filter solo publica dato cuando hay coincidencias. La plantilla usa DefaultValue: 0 y la alarma usa TreatMissingData: notBreaching. Verificá que la instancia esté generando ERROR y esperá un par de períodos de 60 s.

Los logs de EC2 no aparecen en el log group

Síntoma: el log group existe pero está vacío.

Solución: casi siempre es el agente. Entrá por SSM y revisá sudo cat /opt/aws/amazon-cloudwatch-agent/logs/amazon-cloudwatch-agent.log y que el archivo /var/log/app/app.log exista. Confirmá que el rol de la instancia tenga CloudWatchAgentServerPolicy.

El FilterPattern no cuenta lo que espero

Síntoma: la métrica queda en 0 aunque hay errores.

Solución: la sintaxis de patrones distingue mayúsculas/minúsculas. ERROR no matchea error. Ajustá el FilterPattern del metric filter y volvé a desplegar.

El stack queda en ROLLBACK_COMPLETE

Síntoma: la creación falla y se revierte.

Solución: abrí la pestaña Events y buscá el primer error. Causas típicas: no aceptar CAPABILITY_NAMED_IAM, un email con formato inválido, o (con KMS) una key policy que no autoriza al servicio de logs.

Limpieza

Eliminar el stack

CloudFormation› Stacks › obs-clase-2 › Delete

Borrar el stack elimina la instancia EC2, el log group (con sus logs), el metric filter, la alarma, la consulta guardada y, si los creaste, el topic SNS y la KMS key. No hay DeletionPolicy: Retain, así que no queda nada facturable.

bash · eliminar por CLI
aws cloudformation delete-stack --stack-name obs-clase-2 --region us-east-1
KMSSi habilitaste el cifrado, la KMS key entra en estado Pending deletion (ventana de espera por defecto). No genera costo relevante y se elimina sola al vencer el plazo.
Checkpoint
El stack desaparece de la lista (estado DELETE_COMPLETE) y el log group ya no aparece en CloudWatch Logs.
